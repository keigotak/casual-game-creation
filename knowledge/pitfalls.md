# 避けるべき落とし穴

> プロジェクトで遭遇した問題と回避策を記録します。
> 同じ失敗を繰り返さないための参考資料です。

## コード構造

### 単一ファイルの肥大化
**問題**: `Game.jsx` が2000行超えて管理困難に（Dinosaviver）
**原因**: 最初から分割を考慮しなかった
**回避策**:
- 500行を目安に分割検討
- ゲームロジックは最初から `src/game/` に分離
- `/design-gdd` でアーキテクチャも計画

### 定数のハードコード
**問題**: マジックナンバーが散在して調整困難
**原因**: 「後で直す」が積み重なった
**回避策**:
```javascript
// Bad
if (player.hp <= 0) { ... }
const damage = 10 * level;

// Good
const GAME_CONFIG = {
  PLAYER_DEATH_THRESHOLD: 0,
  BASE_DAMAGE: 10,
};
```

### テストの欠如
**問題**: リファクタリングや機能追加時にデグレが発生
**原因**: 時間的制約でテストを省略
**回避策**:
- `src/game/` の純粋関数は必ずテストを書く
- vitestを初期セットアップに含める（`/init-project` で対応済み）
- 最低限：config値テスト、スコア計算テスト、衝突判定テスト

## 国際化（i18n）

### 翻訳キー追加忘れ
**問題**: 新機能追加時に翻訳キーを忘れる
**原因**: 手動チェックに依存
**回避策**:
- `/i18n-check` を実装後に必ず実行
- 新しいテキストは最初から `t('key')` で書く

### 言語ファイルの不整合
**問題**: `ja.json` にあって `en.json` にないキー
**回避策**:
- `/i18n-check` でキー差分チェック
- `/translate all` で全言語を同時更新

## パフォーマンス

### 不要な再レンダリング
**問題**: 毎フレーム全コンポーネントが再レンダリング
**原因**: 状態管理の粒度が粗い
**回避策**:
```javascript
// Bad - 1つの巨大state
const [game, setGame] = useState({ player: {}, enemies: [], ... });

// Good - 分割したstate
const [player, setPlayer] = useState({});
const [enemies, setEnemies] = useState([]);
```

### メモリリーク
**問題**: ゲーム終了後もインターバルが動き続ける
**回避策**:
```javascript
useEffect(() => {
  const interval = setInterval(gameLoop, 16);
  return () => clearInterval(interval);  // クリーンアップ必須
}, []);
```
**チェック**: `requestAnimationFrame`, `setInterval`, `addEventListener` は必ず解除。

### new Audio() のモバイル問題
**問題**: モバイルで効果音が遅延する、再生されない
**原因**: HTMLAudioElementは再生制限が厳しい
**回避策**:
- Web Audio API (`AudioContext`) ベースの SoundManager を使う
- `knowledge/reusable-code/sound-manager.js` を参照
- 初回ユーザー操作時に `AudioContext` を初期化する

## Git/リリース

### コミット粒度が大きすぎる
**問題**: 「全部実装した」の1コミットで履歴が追えない
**回避策**:
- 機能単位でコミット
- `/implement-feature` 使用時に適切な粒度を意識

### .env や認証情報のコミット
**問題**: APIキーがリポジトリに...
**回避策**:
- `.gitignore` を最初に設定（`/init-project` で対応済み）
- `/release` で警告チェック

### itch.ioで画面が真っ白
**問題**: ローカルでは動くがitch.ioで真っ白
**原因**: `vite.config.js` の `base` が `/` のまま
**回避策**:
- `base: './'` を必ず設定（`/init-project` で対応済み）
- アセットパスが絶対パスになっていないか確認

## ゲームデザイン

### 初回体験が悪い
**問題**: 遊び方が分からずにユーザーが離脱
**原因**: チュートリアルが長い、または説明がない
**回避策**:
- 3秒でゲームの種類が分かるビジュアル
- 10秒で操作を理解できる設計（説明テキスト不要が理想）
- `/playtest-review` で初回体験を重点チェック

### リトライの摩擦
**問題**: ゲームオーバー後のリトライが面倒でやめてしまう
**原因**: 画面遷移が多い、ロード時間がある
**回避策**:
- 1タップ/1クリックでリトライ可能に
- ゲームオーバー画面にリトライボタンを最も目立つ位置に配置

### 難易度が急すぎる
**問題**: 突然難しくなって理不尽に感じる
**回避策**:
- 難易度パラメータを config に定義し、曲線で制御
- 「もう1回やれば行けそう」と思える調整が目標
- `/playtest-review` で難易度曲線をチェック
