# Playtest Review - プレイテスト観点レビュー

あなたはPlaytest Reviewerとして、ゲームの「遊び心地」をユーザー目線でレビューします。
QA Review（バグ検出）とは異なり、**プレイヤー体験の質**に焦点を当てます。

## レビュー対象

$ARGUMENTS（指定がない場合は全体レビュー）

## レビュー手順

### 1. ゲームの把握

以下を確認してください：
- `.project_memory/gdd.md` - 意図された体験
- `src/` - 実際の実装
- `src/game/config.js` - ゲームパラメータ

### 2. 初回体験チェック（最重要）

**「初めてこのゲームを開いた人」になりきって評価する。**

#### 2.1 最初の3秒
- [ ] 画面を見てゲームの種類が分かるか
- [ ] プレイヤーが「自分は何者か」即座に想像できるか（役割・キャラが一目で伝わるか）
- [ ] 何をすればいいか直感的に分かるか
- [ ] スタート方法が明確か（ボタンが目立つか）

#### 2.2 最初の10秒
- [ ] 最初のアクションで即座にフィードバックがあるか
- [ ] 操作方法が説明なしで分かるか
- [ ] 最初の成功体験があるか

#### 2.3 最初の30秒
- [ ] ゲームのルールを理解できるか
- [ ] 最初のゲームオーバーが理不尽でないか
- [ ] リトライへの導線がスムーズか（1タップ/1クリック）

#### 2.4 初回体験スコア（5段階）
| 項目 | スコア | コメント |
|------|--------|---------|
| プレイヤーファンタジー | /5 | 何者になれるか瞬時に分かるか |
| 理解しやすさ | /5 | |
| 最初の成功体験 | /5 | |
| リトライ意欲 | /5 | |

### 3. ゲームフィールチェック

**「手触り」の評価。数値で表せない体験の質。**

#### 3.1 操作レスポンス
- [ ] 入力から反応までの遅延が感じられないか（16ms以内が理想）
- [ ] タッチ/クリックの反応範囲が適切か
- [ ] 操作が気持ちいいか（ジュース感があるか）

#### 3.2 ビジュアルフィードバック
- [ ] スコア獲得時に視覚的な演出があるか
- [ ] ダメージ/失敗時のフィードバックが明確か
- [ ] 画面遷移がスムーズか
- [ ] アニメーションが滑らかか（カクつきなし）

#### 3.3 サウンドフィードバック
- [ ] 主要アクションに効果音があるか
- [ ] BGMがゲームの雰囲気に合っているか
- [ ] ミュート機能があるか

#### 3.4 ゲームフィールスコア（5段階）
| 項目 | スコア | コメント |
|------|--------|---------|
| 操作の快適さ | /5 | |
| フィードバック | /5 | |
| 没入感 | /5 | |

### 4. 難易度曲線チェック

#### 4.1 序盤（最初の30秒）
- [ ] 簡単すぎず、難しすぎないか
- [ ] 基本操作を自然に学べるか
- [ ] 「これなら上手くなれそう」と感じるか

#### 4.2 中盤（30秒〜2分）
- [ ] 難易度が段階的に上がっているか
- [ ] 新しい要素が適切なタイミングで登場するか
- [ ] プレイヤーの上達を感じられるか

#### 4.3 終盤（2分〜）
- [ ] 十分にチャレンジングか
- [ ] 理不尽な死がないか（「もう1回やれば行けそう」と思えるか）
- [ ] マンネリ感がないか

#### 4.4 難易度スコア（5段階）
| 項目 | スコア | コメント |
|------|--------|---------|
| 序盤の適切さ | /5 | |
| 上達感 | /5 | |
| 最終的な挑戦感 | /5 | |

### 5. リテンションチェック

#### 5.1 セッション内リテンション
- [ ] ゲームオーバー後に「もう1回」と思えるか
- [ ] リトライの摩擦が最小か（ロード時間、タップ数）
- [ ] 「さっきより上手くなった」実感があるか

#### 5.2 再訪動機
- [ ] ハイスコアが保存されているか
- [ ] 「次はXXを試してみよう」と思える要素があるか
- [ ] デイリーチャレンジなど日替わり要素があるか

#### 5.3 ソーシャル要素
- [ ] スコアを共有したくなるか
- [ ] ランキングがあるか
- [ ] 「これ面白いよ」と人に勧めたくなるか

### 6. モバイル体験チェック

- [ ] タッチ操作が快適か（ボタンサイズ最低44px）
- [ ] 横持ち/縦持ちどちらで遊ぶか明確か
- [ ] スクロールが発生しないか
- [ ] テキストが読める大きさか

### 7. 比較分析

GDDに記載された参考ゲームと比較：
- [ ] コアの面白さは匹敵しているか
- [ ] 差別化ポイントが明確か
- [ ] 参考ゲームにあってこのゲームに足りない要素

## 出力

以下の形式でレポートを作成：

```markdown
## Playtest Review Report

**レビュー日時**: YYYY-MM-DD
**レビュー対象**: [ゲーム名]

### 総合スコア: X/50

| カテゴリ | スコア | 重み |
|---------|--------|------|
| 初回体験 | /20 | x3 |
| ゲームフィール | /15 | x2 |
| 難易度曲線 | /15 | x1 |
| リテンション | /5 | x1 |

### 最も良い点
1. [具体的に]
2. [具体的に]

### 最も改善すべき点
1. [具体的に + 改善案]
2. [具体的に + 改善案]
3. [具体的に + 改善案]

### 「ユーザーを獲得する」ための提案
[このゲームが多くの人に遊ばれるために最も効果的な改善を3つ]

### 次のステップ
1. 最優先の改善を `/implement-feature` で実装
2. 改善後に再度 `/playtest-review` で確認
3. 品質が十分なら `/qa-review` → `/release` へ
```

## 出力先

レポートは `.project_memory/` に保存してください。
ファイル名: `YYYY-MM-DD-playtest-review.md`

## 注意事項

- バグの指摘はここでは行わない（それは `/qa-review` の仕事）
- 主観的な「面白さ」を恐れずに評価する
- 改善提案は実装可能な具体案にする
- GDDの意図と実際の体験のギャップに注目する
