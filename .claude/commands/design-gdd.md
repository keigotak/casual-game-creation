# Design GDD - ゲームデザインドキュメント作成/更新

あなたはDesigner Agentとして、ゲームデザインドキュメント（GDD）を作成・更新します。

## 対象

$ARGUMENTS（新規作成の場合はゲームコンセプトを指定）

## 手順

### 1. 現在のGDD確認

`.project_memory/gdd.md` を読み、現在の状態を把握してください。

### 2. ユーザー要求の分析

ユーザーの要求を分析し、以下を特定してください：
- 新規作成か更新か
- 変更が必要なセクション
- 追加が必要な情報

### 3. GDD作成/更新

以下のテンプレート構造に従ってGDDを作成してください。新規の場合は全セクション、更新の場合は該当セクションのみ。

---

## GDDテンプレート構造

### Section 1: ゲーム概要

```markdown
## 1. ゲーム概要

**タイトル**: [仮タイトル]
**ジャンル**: [ジャンル]
**プラットフォーム**: Web（デスクトップ + モバイル）
**ターゲット**: [誰が、どんな時に遊ぶか]
**1行コンセプト**: [1文でゲームの魅力を説明]
**参考ゲーム**: [似たゲーム、インスピレーション元]
```

### Section 2: コアループ（最重要）

**ゲームの心臓部。これが面白くなければ何も始まらない。**

```markdown
## 2. コアループ

### 2.1 基本サイクル
[アクション] → [フィードバック] → [報酬] → [動機] → [アクション]

例:
タップ → ボールが跳ねる → 障害物を越える → スコア+1 → もっと先へ行きたい → タップ

### 2.2 1プレイの長さ
- 目標: [30秒〜3分]
- 最短（初心者）: [10秒]
- 最長（上級者）: [5分]

### 2.3 リプレイ動機
- [ ] スコア更新への挑戦
- [ ] ランダム要素による変化
- [ ] アンロック要素
- [ ] デイリーチャレンジ
- [ ] ソーシャル（ランキング）
```

### Section 3: 初回体験設計（ユーザー獲得の鍵）

**最初の10秒で「面白い」と思わせないとユーザーは離脱する。**

```markdown
## 3. 初回体験（First Time User Experience）

### 3.1 最初の3秒
- 画面に何が見えるか: [具体的に]
- プレイヤーが理解すべきこと: [1つだけ]
- 行動を促すUI: [大きなスタートボタン / タップで開始]

### 3.2 最初の10秒（チュートリアル不要が理想）
- 最初のアクション: [何をさせるか]
- 最初の成功体験: [何が起きるか]
- 最初のフィードバック: [視覚・聴覚・触覚]

### 3.3 最初の30秒
- ルール理解完了の目安: [どこまで理解できればOK]
- 最初の失敗: [ゲームオーバーの体験]
- リトライへの導線: [1タップでリトライ可能]

### 3.4 2回目のプレイ
- 「次はこうしよう」と思える要素: [具体的に]
- 前回より上手くなれた実感: [何で伝えるか]
```

### Section 4: ゲームプレイ

```markdown
## 4. ゲームプレイ

### 4.1 操作方法
**デスクトップ**:
| キー/操作 | アクション |
|----------|----------|
| [キー] | [アクション] |

**モバイル**:
| 操作 | アクション |
|------|----------|
| タップ | [アクション] |

### 4.2 ゲームメカニクス
[各メカニクスの詳細。数値は具体的に]

### 4.3 ゲームオブジェクト
| オブジェクト | 役割 | パラメータ |
|------------|------|----------|
| プレイヤー | [役割] | 速度: Xpx/s, サイズ: Xpx |
| 障害物 | [役割] | 間隔: Xpx, 速度: Xpx/s |

### 4.4 スコアリング
| アクション | スコア |
|----------|--------|
| [アクション] | +X点 |
```

### Section 5: 難易度設計

```markdown
## 5. 難易度設計

### 5.1 難易度曲線
| 時間/スコア | 難易度 | 変化する要素 |
|-----------|--------|------------|
| 0〜10秒 | 極易 | 障害物少ない、速度遅い |
| 10〜30秒 | 易 | 障害物増加 |
| 30秒〜1分 | 普通 | 速度上昇 |
| 1分〜2分 | 難 | 新パターン登場 |
| 2分〜 | 極難 | 全要素がMAX |

### 5.2 難易度パラメータ
```javascript
const DIFFICULTY = {
  speed: { initial: 2, max: 8, increaseRate: 0.1 },
  spawnRate: { initial: 2000, min: 500, decreaseRate: 50 },
  // ...
};
```

### 5.3 緩急のデザイン
- 緊張: [どんな場面で]
- 緩和: [どんな場面で]
- 山場: [いつ訪れるか]
```

### Section 6: リテンション設計

```markdown
## 6. リテンション設計

### 6.1 短期リテンション（セッション内）
- 「もう1回」のトリガー: [具体的に]
- ゲームオーバー→リトライの摩擦: [最小限に]

### 6.2 中期リテンション（日をまたぐ）
- [ ] ハイスコア記録
- [ ] デイリーチャレンジ
- [ ] アンロック要素（実績、キャラ、スキン）
- [ ] 段階的な目標提示

### 6.3 長期リテンション（週〜月）
- [ ] ランキング（ローカル / オンライン）
- [ ] シーズン要素
- [ ] コンテンツ追加の余地
```

## 設計原則

GDDを書く際は以下の原則を守る：

1. **シンプルさ優先**: 複雑なシステムより直感的なゲームプレイ。ルールは3つ以内。
2. **具体性**: 数値は必ず具体的に書く（「速い」→「200px/秒」）
3. **初回体験を最優先**: 最初の10秒の設計に最も時間をかける
4. **フィードバック重視**: 全てのプレイヤーアクションに即座の反応
5. **段階的難易度**: 最初は必ず簡単。上級者は自然に高難度へ
6. **リプレイ性**: 1プレイ短く、何度もやりたくなる設計
7. **モバイルファースト**: タッチ操作で快適に遊べること

## 整合性チェック

GDD完成時に以下を確認：
- [ ] コアループが明確か（1文で説明できるか）
- [ ] 初回体験の3秒・10秒・30秒が具体的か
- [ ] 数値バランスが妥当か
- [ ] 操作方法がシンプルか（3アクション以内）
- [ ] リテンション要素が最低2つあるか
- [ ] 実装可能な仕様か（Web技術で実現可能）

## 出力

更新したGDDを `.project_memory/gdd.md` に保存してください。

変更サマリーを以下の形式で報告：

```markdown
## GDD更新サマリー

**更新日時**: YYYY-MM-DD
**更新タイプ**: 新規作成 / セクション追加 / 修正

### 変更内容
- [変更したセクション]: [変更内容の要約]

### コアループ
[1文でコアループを説明]

### 次のステップ
1. GDDの内容を確認・承認
2. `/implement-feature [最初に実装する機能]` で実装開始
3. 実装中に仕様変更が必要になったら `/design-gdd update` で更新
```
