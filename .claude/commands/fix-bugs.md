# Fix Bugs - バグ修正

あなたはDeveloper Agentとして、報告されたバグを修正します。

## 対象

$ARGUMENTS（バグID、ファイル名、または「all」で未解決バグ全件）

## 修正手順

### 1. バグ情報の確認

以下を確認してバグの全体像を把握：

```bash
# 未解決バグの一覧
ls .project_memory/bugs/open/
```

対象のバグレポートを読み、以下を整理：
- 深刻度（Critical > High > Medium > Low）
- 再現手順
- 期待される動作 vs 実際の動作
- 影響ファイル

### 2. 優先順位

複数のバグがある場合、以下の順で対応：

1. **Critical**: ゲームが動かない、データが消える
2. **High**: 主要機能が壊れている
3. **Medium**: 見た目やUXの問題
4. **Low**: 細かい改善

### 3. 原因調査

影響ファイルを読み、バグの原因を特定する。

```markdown
### バグ分析: [BUG-ID]
**原因**: [根本原因の説明]
**影響範囲**: [このバグが他にどこに影響するか]
**修正方針**: [どう直すか]
```

### 4. 修正実装

- 最小限の変更で修正する（関連リファクタリングは別タスク）
- 修正と同時に該当箇所のテストを追加/修正

### 5. 検証

```bash
# ビルド確認
npm run build

# テスト実行
npm run test:run
```

修正後、バグレポートの再現手順を実行して問題が解消されたことを確認。

### 6. バグレポートの更新

修正したバグのレポートを `open/` から `closed/` に移動：

```bash
mv .project_memory/bugs/open/[file] .project_memory/bugs/closed/[file]
```

移動したファイルに修正情報を追記：

```markdown
### 修正情報
- **修正日**: YYYY-MM-DD
- **修正ファイル**: [ファイルリスト]
- **修正内容**: [概要]
- **テスト**: [追加/修正したテスト]
```

## 出力

```markdown
## Bug Fix Complete

**日時**: YYYY-MM-DD

### 修正したバグ
| ID | 深刻度 | タイトル | ステータス |
|----|--------|---------|----------|
| BUG-001 | High | [タイトル] | Fixed |
| BUG-002 | Medium | [タイトル] | Fixed |

### 未修正のバグ
| ID | 深刻度 | タイトル | 理由 |
|----|--------|---------|------|
| BUG-003 | Low | [タイトル] | 仕様変更が必要 |

### 検証結果
- [x] ビルド成功
- [x] テストパス
- [x] バグ再現手順で問題解消確認

### 次のステップ
1. `npm run dev` で動作確認
2. 未修正バグがあれば `/design-gdd` で仕様確認後に対応
3. 全バグ修正後は `/playtest-review` で体験確認
4. 問題なければ `/qa-review` で最終チェック
```

## 注意事項

- 修正は最小限に。ついでのリファクタリングはしない
- Critical バグは最優先で対応
- 仕様の問題に起因するバグは、修正前に `/design-gdd` で仕様を確認
- バグ修正で新たなバグを生まないよう、テストで確認
